Cropsy Studio Scripting Documentation

This page explains the current beginner-friendly scripting systems in Cropsy Studio. Right now there are two scripting paths: the command script system used by .crop files, and the new built-in behavior system used in play mode.

Simple on purpose: small API, easy names, safe starting point.

Overview

Command Script

Text-based .crop scripts that run simple scene commands like moving, rotating, scaling, printing, and renaming.

Behavior System

Built-in C++ behaviors that run in play mode with OnStart() and OnUpdate(deltaTime).

Best Use Right Now

Use command scripts for quick scene edits. Use behaviors for repeatable runtime logic.

Quick Start For Beginners

  1. Create or select a simple object like a cube.
  2. Open the Script Editor.
  3. Write a tiny command script first, not a complicated one.
  4. Use Debug before you use Run.
  5. Watch the Console after every test.
Best first goal: make one object move, rotate, or rename itself successfully. Small wins help a lot more than giant scripts. 
# Good first script
print Hello from Cropsy
select Cube
move 0 1 0

Beginner Roadmap

Step 1

Learn names and selection. Make sure you understand what object your script is changing.

Step 2

Practice move, rotate, and set_color.

Step 3

Start using behavior hooks like SpinBehavior in play mode.

If you are new, learn in this order: selection, position, rotation, scale, color, then behaviors.

Command Script Basics

Command scripts are saved as .crop files. They are line-based. Each line is a command. Empty lines are ignored. Comment lines starting with # are ignored too.

A command script does not expose the whole engine. It only supports a small set of safe commands. 
# This is a comment
print Hello from Cropsy
select Cube
move 0 2 0
rotate 0 25 0
set_color 1 0.4 0.2 1

Supported Commands

Command Arguments What It Does
select string Selects a target entity by name. You can also use selected to keep the current one.
print string Writes an info message to the console.
warn string Writes a warning message to the console.
error string Writes an error message to the console.
set_position vec3 Sets the selected entity position directly.
move vec3 Adds the vector to the current position.
set_rotation vec3 Sets Euler rotation directly.
rotate vec3 Adds to the current Euler rotation.
set_scale vec3 Sets scale directly. Very small values are clamped so scale does not become zero.
scale vec3 Multiplies the current scale by the given vector.
set_color vec4 or vec3 Sets mesh tint color. Works only on entities with a mesh component.
set_name string Renames the current target entity.

Examples

Move a cube upward

select Cube
move 0 3 0

Rotate a light

select Light
rotate 0 45 0

Set a sphere to bright orange

select Sphere
set_color 1 0.45 0.15 1

Rename the selected object

set_name MyCrate

Make something spin slowly

select Cube
rotate 0 15 0

Set a direct position instead of adding movement

select Cube
set_position 0 5 0

Scale an object up a little

select Cube
scale 1.1 1.1 1.1

Built-in Behaviors

Behaviors are different from command scripts. They run during play mode and update every frame. Right now the system is intentionally small.

Behavior Name What It Does Where To Use It
SpinBehavior Rotates the object around the Y axis over time. Set Behavior Name in the Inspector.
Unknown behavior names are ignored safely. They do not crash the engine, but they also do nothing.

Behavior API

This is the beginner-friendly API currently exposed by the base Behavior class.

Function Return Type What It Does
EntityId Finds a direct child by name.
bool Checks whether the behavior component is enabled.
string Gets the entity name.
vec3 Gets local position.
vec3 Gets local Euler rotation.
vec3 Gets local scale.
void Runs once when play mode starts.
void Runs every frame during play mode.
void Changes the entity name.
void Sets local position.
void Sets local Euler rotation.
void Sets local scale.
void Turns the behavior component on or off.
void Writes a message to the Cropsy console.

Example Behavior

class SpinBehavior : public Behavior
{
public:
    void OnUpdate(float deltaTime) override
    {
        glm::vec3 rotation = GetRotation();
        rotation.y += 60.0f * deltaTime;
        SetRotation(rotation);
    }
};

Common Mistakes

Mistake Why It Happens Better Approach
Forgetting to select a target The script does not know what object to edit. Use select Cube first.
Using the wrong object name The name in the hierarchy does not match the script text. Copy the exact name from the hierarchy.
Mixing up move and set_position One adds movement, the other replaces the full position. Use move for offsets and set_position for exact placement.
Typing too much before testing Big scripts are harder to debug. Run tiny changes often.

Debugging Tips

  • Use print often while learning.
  • Use the Debug button before Run if you are unsure.
  • If a command does nothing, check the object name first.
  • If color fails, make sure the target has a mesh component.
  • If a behavior does nothing, make sure the behavior name matches exactly.
Good debugging habit: change one thing, test it, then change the next thing.

Practice Ideas

Practice 1

Make a cube move upward and then rename it.

Practice 2

Make a sphere orange and slightly bigger.

Practice 3

Assign SpinBehavior to one object and compare it with a command script.

If you want to improve quickly, repeat small experiments until you can predict what the script will do before you run it.

Current Limits

The command script system is intentionally small. Unsupported commands produce a script error instead of trying to guess what you meant.
  • Cropsy Studio uses C++20, but full C++20 support is not implemented yet. >> isocpp.org. <<
  • No full custom scripting language.
  • Behavior names are hand-written built-ins right now.
  • Command scripts work best for simple scene edits and debug tasks.
If you add a behavior name that does not exist, the engine safely ignores it. That is normal right now.
